Observación · Contexto

El problema con YouTube Analytics

El dashboard nativo de YouTube no muestra los datos con el nivel de detalle necesario.

Lo que YouTube Analytics muestra
  • ✕Crecimiento de vistas solo semana actual vs anterior
  • ✕Sin histórico acumulado configurable
  • ✕Sin contexto por playlist
  • ✕Sin vista comparativa semana a semana a lo largo del tiempo
Lo que este proyecto responde
  • ✓Histórico acumulado con filtro de fechas libre
  • ✓Rendimiento por playlist y video semana a semana
  • ✓Evolución comparativa de múltiples playlists
  • ✓Datos propios, controlados, en cualquier formato
La oportunidad: construir una herramienta propia de análisis usando la YouTube Data API v3, con infraestructura serverless de bajo costo, que responda exactamente las preguntas de negocio que importan para entender el canal.
Planteamiento del Problema · Objetivos

Preguntas de negocio que guían el proyecto

El enfoque no es hacer Business Intelligence con un modelo relacional, sino escribir queries SQL que respondan preguntas concretas del canal.

Canal
  • ▸¿Cómo ha crecido el número de suscriptores mes a mes?
  • ▸¿Cuál es la tendencia acumulada de vistas del canal?
  • ▸¿Qué tan rápido crece un video nuevo en sus primeras semanas?
Videos & Playlists
  • ▸¿Cuáles son los videos con mayor crecimiento de vistas semana a semana?
  • ▸¿Qué playlist acumula más engagement en la última semana?
  • ▸¿Qué videos entraron o salieron de una playlist recientemente?
◆
Proceso real: primero se define qué se quiere entender del canal, luego se traduce a una query SQL, y finalmente se decide cómo visualizarlo. De la pregunta al dato, del dato al gráfico.

Análisis de Datos · Repo 1

Extracción desde YouTube API v3

YouTubeClient encapsula todas las llamadas a la API. Cada método corresponde a un endpoint distinto. 

from googleapiclient.discovery import build

class YouTubeClient:
    def __init__(self):
        self.youtube = build("youtube", "v3", developerKey=api_key)

    def get_channel_stats(self):           # → suscriptores, vistas, videos
        return self.youtube.channels().list(part="statistics", id=channel_id)

    def get_recent_video_ids(self):         # → últimos 5 videos publicados
        return self.youtube.search().list(part="snippet", order="date", ...)

    def get_all_video_ids(self):           # → todos los videos via playlist uploads
        return self.youtube.playlistItems().list(playlistId=uploads_id, ...)

    def get_videos_details(self, video_ids): # → métricas por video
        return self.youtube.videos().list(part="snippet,statistics", ...)

    def get_all_playlists(self):            # → playlists manuales del canal
        return self.youtube.playlists().list(channelId=channel_id, ...)
Por qué diario y semanal: los datos de suscriptores y vistas del canal fluctúan diariamente — es útil tenerlos cada día. Los metadatos de todos los videos y playlists no cambian tanto, pero sobre todo: hacer todo diario aumentaría las llamadas a la API, que es el recurso más limitado y costoso. La división minimiza el consumo de cuota de API y permite escalar el canal sin incurrir en costos adicionales.

Recomendaciones · Puntos Críticos

Lo que hay que tener en cuenta para mantener el pipeline

!
El semanal no puede romperse
Las tablas de tipo snapshot acumulan histórico semana a semana. Si el scheduler falla y no se recupera, se pierde esa semana del historial. El histórico no se puede reconstruir: es irrecuperable.
!
Separar lo que puede cambiarse de lo que no
Se puede editar: las queries SQL del Repo 2 (vistas analíticas, nuevas columnas calculadas, filtros).
No se puede editar sin cuidado: el código que construye los snapshots en el Repo 1. Cambiar la estructura de esas tablas sin migración puede romper todo el histórico acumulado.
▸
Las definiciones iniciales son contratos
El esquema de las tablas de snapshot, la granularidad y las claves de idempotencia están definidas desde el inicio. Agregar un nuevo análisis es fácil. Cambiar la estructura de una tabla existente requiere migración y puede afectar a los datos históricos ya acumulados.
▸
Monitorizar el Cloud Scheduler y las Cloud Functions
Los schedulers están configurados con reintentos automáticos, pero conviene revisar periódicamente los logs en Cloud Functions para detectar fallos silenciosos, especialmente si se hacen cambios en las queries o en las dependencias del proyecto.
